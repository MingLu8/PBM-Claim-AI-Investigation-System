Agentic AI in 3 Days — using the Dads .NET backend 🤖

Welcome! You already have a working agentic AI service — the brain behind the Dads chat assistant. It's an ASP.NET Core minimal-API app (this repo, PBM-Claim-AI-Investigation-System, project src/ApiGateway) that wraps a Large Language Model, gives it tools it can call, a memory it can search, and a fallback model when the cloud one fails. Same superpower as the React course: you're learning agents on a real app you can run, not a toy.

Who this is for: .NET developers who know C# and REST APIs and want to understand how an LLM agent is actually built — no prior AI experience needed.

How it works: 3 core days, plus a bonus unit on local models (Day 3½) and a hands-on build-it-yourself appendix. Every concept is shown in our code, with a short 🔌 plain chatbot → 🤖 agent translation and a 🛠️ Try it challenge. Do the challenges — that's where it sticks.

The golden rule of agentic AI, up front:

An agent is just an LLM in a loop that you let call your functions. You describe the tools; the model decides which to call, reads the results, and keeps going until it can answer. That's 80% of it. The other 20% is plumbing: messages in, tools registered, results fed back.

Two libraries do the heavy lifting (both Microsoft, both already referenced):

  • Microsoft.Extensions.AI (MEAI) — a vendor-neutral IChatClient abstraction. "Talk to an LLM" without caring whether it's Gemini, OpenAI, or a local model.
  • Microsoft.Agents.AI — the ChatClientAgent that wraps an IChatClient with instructions + tools and runs the tool-calling loop for you.

Day 1 — The chat client & the agent loop

Goal: understand the universal LLM interface (IChatClient), how messages/roles work, and how ChatClientAgent turns "one model call" into an autonomous loop.

1.1 IChatClient — the universal LLM interface

Everything starts with one MEAI interface. You hand it messages; it returns a response:

Task<ChatResponse> GetResponseAsync(
    IEnumerable<ChatMessage> messages,
    ChatOptions? options = null,
    CancellationToken cancellationToken = default);

That's the seam that makes the whole app vendor-neutral: Gemini, OpenAI, or a local Ollama model all implement the same IChatClient, so the agent above doesn't care which one it's talking to.

🔌→🤖 The old way: a GeminiService with a bespoke AskGemini(string) method, tightly coupled to one vendor's SDK. The agentic way: program against IChatClient; swap providers by swapping the registration in DI.

1.2 Messages & roles — the conversation is a list

An LLM is stateless: every call you send the whole conversation as a list of ChatMessages, each tagged with a role:

RoleWhoExample in our app
Systeminstructions / persona"You are a PBM assistant… always call the memory tool first."
Userthe human"What is my member ID in this claim?"
Assistantthe modelthe reply, or a request to call a tool
Toolyour code's answerthe JSON a tool returned, fed back to the model

Each message carries Contents — not just text, but typed parts like TextContent, FunctionCallContent (the model wants to call a tool), and FunctionResultContent (your tool's output). That typed content is what makes tool-calling possible (Day 2).

1.3 GeminiChatClient — a real client over REST

We didn't use a vendor SDK; GeminiChatClient.cs implements IChatClient directly against Gemini's HTTP API. The shape:

public async Task<ChatResponse> GetResponseAsync(
    IEnumerable<ChatMessage> messages, ChatOptions? options = null, CancellationToken ct = default)
{
    var url = $"https://generativelanguage.googleapis.com/v1beta/models/{_model}:generateContent";

    // Gemini takes the persona in a dedicated "systemInstruction" field, not as a chat turn.
    var systemParts = new List<string>();
    if (!string.IsNullOrWhiteSpace(options?.Instructions))     // ← the agent's persona (see deep dive)
        systemParts.Add(options.Instructions);
    systemParts.AddRange(messages.Where(m => m.Role == ChatRole.System).Select(m => m.Text)...);

    var requestBody = new GeminiRequest {
        Contents          = BuildContents(messages),           // user/assistant/tool turns → Gemini "contents"
        SystemInstruction = ...,                               // the merged system text
        Tools             = BuildTools(options)                // your tools → Gemini function declarations
    };
    // POST with x-goog-api-key header, then map the JSON back into a ChatResponse.
}

The interesting work is the mapping: BuildContents converts MEAI messages (including the tool-call loop) into Gemini's format, and BuildChatResponse converts Gemini's reply back into a ChatResponse with either text or FunctionCallContent. You rarely write this yourself (prebuilt clients exist), but reading it once demystifies "what actually goes over the wire."

1.4 The agent — instructions + tools + a loop

A bare IChatClient answers once. To make it agentic, wrap it in a ChatClientAgent (Microsoft.Agents.AI). From DependencyResolution.cs:

return new ChatClientAgent(
    chatClient,                                  // the IChatClient (Gemini, with Ollama fallback)
    name: "PharmacyParserAgent",
    instructions: """
        You are a PBM (Pharmacy Benefit Management) assistant.
        MEMORY: ALWAYS call 'recall_past_conversations' FIRST when the user refers to a past chat…
        TOOLS: Use extract_pharmacy_npi and extract_cardholder_id to read NCPDP claims…
        """,
    tools: new List<AITool>
    {
        AIFunctionFactory.Create(npiPlugin.ExtractPharmacyNpi,  name: "extract_pharmacy_npi"),
        AIFunctionFactory.Create(cardPlugin.ExtractCardholderId, name: "extract_cardholder_id"),
        AIFunctionFactory.Create(memoryTool.RecallAsync,         name: memoryTool.Definition.FunctionName, ...)
    });

Now a single await agent.RunAsync(messages) may, under the hood, call the model several times: model asks for a tool → the framework runs your C# method → appends the result → calls the model again → repeats until the model returns a final answer. That loop is the agent.

🔌→🤖 Plain chatbot: var reply = await llm.Ask(userText); — one shot, no tools. Agent: var reply = await agent.RunAsync(messages); — the model may call your functions any number of times before answering. You wrote the tools; the model orchestrates them.

💡 Deep dive: what actually makes it "agentic"

The magic isn't the model — it's the loop around it. Concretely, one RunAsync can expand to:

  1. Send [system, …history…, user] + the tool declarations to the model.
  2. Model replies with a FunctionCallContent: "call extract_cardholder_id with this payload."
  3. The framework finds the matching AITool, invokes your C# method, gets a string back.
  4. It appends a FunctionResultContent (role Tool) and calls the model again.
  5. Now the model has the answer and returns final text. Loop ends.

You never wrote an if (intent == "getMemberId") switch. You registered tools and let the model route. That inversion — the model drives control flow — is the whole idea, and also why Day 2 (good tool descriptions) and the gotchas (determinism) matter so much.

💡 Deep dive: where the persona lives — and a real bug

The agent's instructions are not sent as a User message. The framework passes them via ChatOptions.Instructions to the IChatClient on every call. Your client has to honor that — and a custom client can silently forget to.

We hit exactly this. GeminiChatClient originally built its systemInstruction only from System-role messages and ignored options.Instructions — so the whole PBM persona ("you have memory, always call the recall tool…") never reached Gemini. The symptom: the bot answered like a generic "large language model, trained by Google," with no awareness of its tools or memory. The fix is the merge you saw in 1.3:

if (!string.IsNullOrWhiteSpace(options?.Instructions))
    systemParts.Add(options.Instructions);   // ← without this line, the agent's brain is silently dropped

Lesson: when you implement IChatClient yourself, the agent configures behavior through ChatOptions (Instructions, Tools, temperature…). Honor those fields, or the agent above you stops working in ways that look like "the model is dumb" rather than "a field was ignored."

🛠️ Day 1 challenges

  1. In Swagger, POST /api/session then /api/chat with a simple question. Watch the reply come back.

Day 1 recap: IChatClient = talk to any LLM; the conversation is a list of role-tagged messages; ChatClientAgent = client + instructions + tools + the tool-calling loop; the persona travels in ChatOptions.Instructions — honor it.

Day 2 — Tools (function calling)

Goal: give the model hands. A tool is just a C# method the model is allowed to call. Today: how a method becomes a tool, how its "shape" reaches the model, and the call/result round-trip.

2.1 A tool is a C# method + [Description]

Look at PharmacyNpiParser.cs — there is nothing "AI" about it. It's an ordinary method, annotated so the model knows what it does:

public class PharmacyNpiParser
{
    [Description("Parses a raw NCPDP D0 claim string to extract the 10-digit Pharmacy NPI.")]
    public string ExtractPharmacyNpi(
        [Description("The raw NCPDP telecom transaction string")] string ncpdpPayload)
    {
        // …find the "201-B1" segment, slice out the 10 digits, return them…
    }
}

Those [Description] attributes aren't documentation for you — they become part of the prompt the model reads when deciding whether and how to call the tool (more in the deep dive).

2.2 AIFunctionFactory.Create — method → tool

One call turns a method (and its [Description]s and parameter types) into an AITool the agent can expose. That's the registration you saw on Day 1:

AIFunctionFactory.Create(npiPlugin.ExtractPharmacyNpi,  name: "extract_pharmacy_npi")
AIFunctionFactory.Create(cardPlugin.ExtractCardholderId, name: "extract_cardholder_id")
AIFunctionFactory.Create(memoryTool.RecallAsync,         name: "recall_past_conversations", description: ...)

The factory reflects over the method signature to build a JSON schema of its parameters. The model never sees your C#; it sees name + description + parameter schema, and emits a JSON arguments object when it wants to call it.

🔌→🤖 The old way to "route" a request: parse the user text, switch on keywords, call the right service. The agentic way: register tools with good descriptions and let the model pick. Adding a capability = adding a method, not editing a router.

2.3 How the schema reaches the model (and Gemini's quirks)

Each provider wants tools in a slightly different shape. GeminiChatClient.BuildTools converts the agent's AITools into Gemini function declarations, and SanitizeSchema strips JSON-schema keywords Gemini's parser rejects:

// Schema keywords Gemini rejects; removed before sending.
private static readonly HashSet<string> UnsupportedSchemaKeys =
    new(StringComparer.Ordinal) { "$schema", "additionalProperties" };

You don't normally hand-write this, but it's a useful reminder: "function calling" is just structured JSON over the wire, and each model has rules about what JSON it accepts.

2.4 The call/result round-trip

When the model decides to use a tool, the response contains a FunctionCallContent (name + arguments). The agent loop invokes your method, then sends the answer back as a FunctionResultContent. One Gemini wrinkle our client handles: Gemini requires the function response to be a JSON object, so WrapFunctionResult passes objects through and wraps bare strings as { "result": "…" }:

internal static JsonElement WrapFunctionResult(object? result)
{
    var raw = result as string ?? JsonSerializer.Serialize(result);
    try { using var doc = JsonDocument.Parse(raw);
          if (doc.RootElement.ValueKind == JsonValueKind.Object) return doc.RootElement.Clone(); }
    catch (JsonException) { /* not JSON */ }
    return JsonSerializer.SerializeToElement(new { result = raw });   // wrap non-objects
}

💡 Deep dive: tool descriptions ARE prompt — write them for the model

The model chooses a tool entirely from its name, description, and parameter descriptions. Vague text → the tool gets ignored or misused. Compare our memory tool's description (RecallMemoryTool.cs):

"Search THIS user's earlier conversations for relevant facts. Call this when the user references a previous chat, uses an entity or pronoun with no antecedent in the current conversation, or asks to continue earlier work."

Notice it tells the model when to call it, not just what it does. That "call this when…" framing is the single biggest lever on whether tools actually fire. Treat every [Description] as prompt engineering.

💡 Deep dive: keep tools deterministic, cheap, and JSON-shaped

Because the model decides when to call a tool — possibly several times in one turn — good tools are: deterministic (same input → same output; our parsers are pure string slicing), cheap/fast (they sit inside a latency-sensitive loop), and they return strings/JSON the model can read. RecallAsync returns JsonSerializer.Serialize(new { method, count, snippets }) — compact, structured, easy for the model to use. Avoid side effects in tools the model might call speculatively.

Day 2 recap: a tool = a C# method + [Description]; AIFunctionFactory.Create turns it into an AITool with a JSON schema; the model emits a call, the loop runs it and feeds the result back; descriptions are prompt; keep tools deterministic and JSON-shaped.

Day 3 — Memory, RAG & resilience

Goal: make the agent remember across turns and sessions, and keep working when the cloud LLM fails.

3.1 Two kinds of memory

LLMs are stateless, so "memory" is something you provide:

  • In-session history — replay this conversation's earlier turns on every call. Cheap, exact.
  • Cross-session recall — search the user's other conversations for relevant facts and inject them. This needs embeddings + RAG (3.3).

3.2 The orchestration — ChatEndpoint

ChatEndpoint.cs is where a turn is assembled. Read it as four steps:

// 1. Cross-session memory: recall relevant facts and inject them as a System message.
if (memory.IsEnabled) {
    var snippets = await memory.RecallSnippetsAsync(userInput, excludeSessionId: sessionId, token);
    if (snippets.Count > 0)
        messages.Add(new ChatMessage(ChatRole.System, "Relevant facts recalled…\n" + string.Join("\n", snippets)));
}
// 2. Replay THIS session's history for continuity.
foreach (var turn in history)
    messages.Add(new ChatMessage(turn.Role == "assistant" ? ChatRole.Assistant : ChatRole.User, turn.Content));
// 3. The new user message, then run the agent (tools incl. recall remain available to the model).
messages.Add(new ChatMessage(ChatRole.User, userInput));
var response = await agent.RunAsync(messages, cancellationToken: token);
// 4. Persist the exchange so future sessions can recall it.
history.Add(new ChatTurn("user", userInput));
history.Add(new ChatTurn("assistant", response.Text));
await store.SetASync(updated);

So memory is both proactive (step 1 injects facts up front) and agentic (the model can also call recall_past_conversations itself mid-loop). Two paths to the same store.

3.3 Embeddings — turning text into vectors

To find "relevant" past turns you compare meaning, not keywords. An embedding is a vector (array of floats) where similar text lands nearby. Our IEmbedder interface has two real implementations:

  • GeminiEmbedder — calls Gemini's embedContent endpoint. High quality, needs an API key/network.
  • InMemoryEmbedder — a deterministic offline embedder (feature-hashing with FNV-1a, L2-normalized). No network, always available — perfect as a fallback and for tests.

(There's also an AzureOpenAIEmbedder stub, wired but not yet implemented.) Which one runs is chosen in DependencyResolution.cs by the Embeddings:Provider config.

3.4 RAG — rank by similarity, inject the top hits

Retrieval-Augmented Generation = retrieve relevant text, then let the model generate with it in context. InMemoryRagSearch embeds the query and every candidate turn, scores them by cosine similarity, and returns the top K above a threshold:

var queryVector = await embedder.EmbedAsync(query, ct);
foreach (var doc in docs) {
    var score = CosineSimilarity(queryVector, await EmbedCachedAsync(doc, ct));
    if (score >= minScore) hits.Add(new RagHit(doc, score));
}
return hits.OrderByDescending(h => h.Score).Take(topK);

RecallMemoryTool.RecallSnippetsAsync uses this when an embedder is available, and falls back to simple keyword matching when it isn't — so recall degrades gracefully instead of breaking.

🔌→🤖 A plain chatbot forgets everything between requests. An agent gets memory by (a) replaying history and (b) RAG-searching a vector store and injecting the hits — the same pattern used for "chat over your documents."

3.5 Sessions — where history lives

ISessionStore abstracts persistence. DependencyResolution picks Redis (RedisSessionStore) when a connection string is configured, else an in-memory store for local dev. The front end calls /api/session to get an id, then sends it with every /api/chat — that id is what ties a user's turns into one conversation the agent can replay.

💡 Deep dive: the fallback pattern — never fully down

Cloud LLMs fail: rate limits, outages, a bad key. Our agent stays up via the decorator pattern in FallbackChatClient.cs — itself an IChatClient that wraps two others:

public async Task<ChatResponse> GetResponseAsync(IEnumerable<ChatMessage> messages, ChatOptions? o, CancellationToken ct)
{
    try { return await _primary.GetResponseAsync(messages, o, ct); }   // Gemini
    catch (Exception ex) {
        _logger.LogWarning(ex, "'{Primary}' failed — falling back to '{Fallback}'.", _primaryName, _fallbackName);
        return await _fallback.GetResponseAsync(messages, o, ct);      // local Ollama
    }
}

DI composes it so the agent is handed the fallback client without knowing it: Gemini primary, local Ollama secondary. The same idea protects embeddings — FallbackEmbedder picks a working embedder once up front (so a conversation never mixes two vector spaces — see gotcha #5). Because both layers are just IChatClient/IEmbedder, resilience is composition, not rewrites.

Day 3½ — Running a local model with Ollama 🦙

The fallback only helps if something's listening locally. Ollama runs open models on your machine and exposes an OpenAI-style API the MEAI OllamaChatClient can call.

# install Ollama (https://ollama.com), then:
ollama pull llama4        # or any model you like
ollama serve              # serves http://localhost:11434

OllamaSettings.cs holds the endpoint and model name; DependencyResolution builds an OllamaChatClient from them and hands it to the FallbackChatClient as the secondary:

public class OllamaSettings
{
    public string Endpoint { get; init; } = "http://localhost:11434";
    public string Model    { get; init; } = "llama4";
}

Now if Gemini errors, the very same agent — same instructions, same tools — keeps answering from a model on your laptop. (Tool-calling quality varies by local model; smaller models may call tools less reliably than Gemini.)

🔌→🤖 Because the agent targets IChatClient, "use a different brain" is a one-line DI change, not a rewrite. Local models also unlock offline dev and keep sensitive data on-prem.

Day 3½ recap: Ollama serves local models behind an IChatClient; configured via OllamaSettings; wired as the fallback so the agent degrades gracefully instead of going dark.

⚠️ The 8 gotchas that bite every agent builder

  1. Instructions must actually reach the model. They travel in ChatOptions.Instructions; a custom IChatClient that ignores that field silently strips the agent's brain. (Day 1 deep dive — our real bug.)
  2. Tool descriptions are prompt. Vague [Description] = the model won't call the tool, or calls it wrong. Say when to use it.
  3. Each provider has schema rules. Gemini rejects $schema/additionalProperties; we strip them in SanitizeSchema. Tool calling is just structured JSON with provider quirks.
  4. Tool results need the right shape. Gemini requires a JSON object response — WrapFunctionResult wraps bare strings. Returning the wrong shape breaks the loop.
  5. Keep one embedding vector space. Never mix two embedders within a conversation — distances become meaningless. FallbackEmbedder chooses once, up front.
  6. The model drives control flow — so make tools deterministic. It may call a tool zero, one, or many times. Side-effectful or slow tools will hurt; pure/cheap ones shine.
  7. Session id reuse = memory. Same sessionId across requests is what lets the backend replay history; a fresh id every call means an amnesiac agent.
  8. Streaming isn't free. GeminiChatClient.GetStreamingResponseAsync throws NotImplementedException, so streaming paths fall back. Know which methods your client truly supports.

🏗️ Appendix: build the agent yourself, step by step

Want to construct it from scratch? Each step runs and adds one capability. Each links to the day that explains it.

Step 1 — Talk to an LLM through IChatClient

Reference Microsoft.Extensions.AI. Either implement IChatClient over your provider's REST API (like GeminiChatClient) or use a prebuilt one. Goal: await client.GetResponseAsync(messages) returns text. (Day 1)

Step 2 — Wrap it in an agent

Reference Microsoft.Agents.AI. Create a ChatClientAgent with instructions (the persona) and an empty tool list. ✅ agent.RunAsync(messages) answers in character. (Day 1)

Step 3 — Give it a tool

Write a plain method with [Description] on it and its parameters; register it with AIFunctionFactory.Create. ✅ Ask something only the tool can answer and watch it get called. (Day 2)

Step 4 — An orchestration endpoint

Add a minimal-API POST /api/chat that builds the message list (history + new user turn), calls agent.RunAsync, and returns response.Text. ✅ A real chat endpoint. (Day 3)

Step 5 — Memory (embeddings + RAG)

Add an IEmbedder (start with the offline in-memory one), an IRagSearch that ranks past turns by cosine similarity, and inject the top hits as a System message before running the agent. ✅ Cross-session recall. (Day 3)

Step 6 — Resilience

Wrap your primary IChatClient in a FallbackChatClient that catches and retries on a second client (local Ollama). ✅ The agent survives a cloud outage. (Day 3 & 3½)

Step 7 — Sessions & persistence

Add an ISessionStore (in-memory, then Redis) and a /api/session endpoint; thread the sessionId through every chat call. 🎉 You've built the Dads agent. (Day 3)
